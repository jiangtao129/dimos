# /ship — open a PR against `jiangtao129/dimos:main`

This is the **only** sanctioned way to land a change in this fork.
Encodes the 11-step pipeline. Each step lists the exact command. Stop on
any red, do not skip steps.

## Preconditions (verify before step 1)

- Working tree is clean OR all current changes are intended for THIS PR.
- `git remote -v` shows `github` pointing at `jiangtao129/dimos` (and
  optionally `origin` at the corp GitLab + `upstream` at
  `dimensionalOS/dimos`).
- `gh auth status` shows `jiangtao129` logged in via SSH.
- The user gave a clear scope for THIS PR. One focused change only.

## The 11 steps

### Step 1 — Sync the fork's main

Always start a feature branch from a freshly fetched `github/main`, not
from `feat/jiangtao` or local stale state.

```bash
git fetch github
```

If `github/main` is behind `feat/jiangtao` (the corp-tracking branch),
that's expected — the fork only carries reviewed PRs. Do NOT auto-merge
`feat/jiangtao` into `github/main`; that's the user's call.

### Step 2 — Create the feature branch

```bash
BRANCH="<prefix>/<short-slug>"   # e.g. chore/agent-pipeline-bootstrap
git checkout -b "$BRANCH" github/main
```

Prefixes: `feat/`, `fix/`, `refactor/`, `docs/`, `test/`, `chore/`,
`perf/`. Slugs are kebab-case, ≤ 50 chars.

### Step 3 — Make the change

- Edit only what this PR needs.
- No drive-by refactors, no formatting churn outside changed lines
  (ruff-format will rewrite as needed in step 4).
- If the change requires a new pre-commit hook, a new dependency, or a
  schema change — add a one-line justification to the upcoming PR body.

### Step 4 — `bash scripts/verify.sh`

```bash
bash scripts/verify.sh
```

Must end with `verify.sh: ALL OK`. If red:

- Read the failing step. Fix the **code** (or the new feature), not the
  script. `|| true`, removing tests, or weakening asserts is a P0 block.
- Pre-commit auto-fixes are allowed (ruff-format, license header, EOF
  newline) — re-stage them. The pre-commit `check-commit-message` hook
  rejects auto-generated trailers; rewrite the commit message manually.
- Re-run `bash scripts/verify.sh` until green.

### Step 5 — Commit

```bash
git add -A           # only after verify is green
git -c user.name="jiangtao129" \
    -c user.email="jiangtao129@users.noreply.github.com" \
    commit -m "<conventional message>"
```

Commit message rules:

- Conventional commits: `<type>(<scope>): <imperative summary>`
- No `Generated by Claude Code` / `Co-Authored-By: Claude` /
  `🤖 Generated with [Claude Code]` trailers — pre-commit
  `check-commit-message` will reject them.
- One commit per PR is preferred. If you must use multiple commits,
  squash on merge (step 9 already uses `--squash`).

### Step 6 — Push to the fork

```bash
git push -u github HEAD
```

NEVER `--force` here. If the remote rejects (someone else pushed), stop
and ask the user how to proceed.

### Step 7 — Open the PR via REST API (NOT `gh pr create`)

`gh pr create` sometimes hits a GraphQL index race ("No commits between
main and ...") right after a push. The REST endpoint is reliable:

```bash
cat >/tmp/pr-body.md <<'EOF'
## Summary
<2–4 bullets describing what changed and why>

## Test plan
- [x] `bash scripts/verify.sh` ALL OK locally
- [ ] CI `verify` job passes

<one-line justification for any new dependency / hook / schema change>

Closes <issue-id-if-any>
EOF

gh api repos/jiangtao129/dimos/pulls \
    -X POST \
    -f title="<PR title — same as commit message>" \
    -f head="$BRANCH" \
    -f base="main" \
    -F body=@/tmp/pr-body.md
```

Capture the returned `html_url` and `number` for step 8.

### Step 8 — Enable auto-merge (squash + delete branch)

```bash
PR_NUMBER="<from step 7>"
gh pr merge "$PR_NUMBER" --auto --squash --delete-branch \
    --repo jiangtao129/dimos
```

`--auto` waits for required status checks to pass (the `verify` job in
`.github/workflows/verify.yml`), then squash-merges, then deletes the
remote branch. If branch protection is not yet configured, the merge
happens immediately — that is expected for the bootstrap PR only.

### Step 9 — Monitor CI

```bash
gh pr checks "$PR_NUMBER" --repo jiangtao129/dimos
gh run list --repo jiangtao129/dimos --branch "$BRANCH" --limit 5
```

Wait for the `verify` job to finish. Typical: 3–5 min on first run
(no uv cache), 30–90 s on subsequent runs.

If `verify` fails on CI but `bash scripts/verify.sh` passed locally:

- Most common: GitHub Actions environment lacks something the local
  has (e.g. a Python package not in `uv.lock`, a system tool, a
  network reachability). Fix in a follow-up commit on the same branch.
- Less common: a flaky test that happens to fail on the runner. Add it
  to `_KNOWN_FLAKY` in `./conftest.py` ONLY after confirming the bug is
  upstream's, not the change in this PR.

### Step 10 — Wait for Codex review (when enabled)

Once the user enables Codex auto-review on `jiangtao129/dimos` in the
ChatGPT Codex backend, every PR triggers a review within 30–90 s of
opening (or after CI passes, depending on settings). Possible signals:

- `chatgpt-codex-connector[bot]` reaction `+1` on the PR description:
  "looked, no issues". Terminal state.
- `chatgpt-codex-connector[bot]` reaction `eyes`: "I'm reviewing,
  comments coming". Intermediate state.
- A `COMMENTED` review with a P0/P1/P2/P3 body: real findings. Read
  them; if any are valid, fix in a follow-up commit on the same branch
  (the auto-merge re-arms once CI is green again).

If branch protection is set to "require Codex approval" (NOT the
recommended Q5=B mode), wait for the review before merging.

### Step 11 — Verify the merge

```bash
gh pr view "$PR_NUMBER" --repo jiangtao129/dimos --json state,mergedAt,mergeCommit
git fetch github
git log --oneline github/main -3
```

Expected: PR is `MERGED`, the squash commit is at the tip of
`github/main`, and the remote `$BRANCH` is gone (deleted by `--auto`).

Local cleanup:

```bash
git checkout feat/jiangtao    # or wherever your work tree lives
git branch -D "$BRANCH"        # remove the now-merged feature branch
```

## What NOT to do

- Do NOT push directly to `main` (or `dev`).
- Do NOT use `git push --force` on `main` or any reviewed PR branch.
- Do NOT skip `bash scripts/verify.sh` even for "trivial" changes.
- Do NOT click GitHub UI settings (branch protection, Codex toggle,
  secrets) on the user's behalf — emit a TODO checklist instead.
- Do NOT run `gh pr create` and hope the GraphQL race doesn't bite.
  Always REST.
- Do NOT batch unrelated changes into one PR.
- Do NOT modify `AGENTS.md` content above the
  `## Pipeline (fork @ \`jiangtao129/dimos\`)` anchor.

## Bootstrap PR exception

The very first PR (`chore/agent-pipeline-bootstrap`) is opened BEFORE
branch protection exists. It will auto-merge immediately on `--auto`
because there are no required checks yet. That is intentional and is
the only PR that is allowed to merge with no enforcement. After it
lands, the user enables branch protection (see step 11 of the bootstrap
TODO list emitted by the agent).
